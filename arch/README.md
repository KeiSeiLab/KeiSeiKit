# `arch/` — Self-Validating Architecture

This directory is the SSoT for KeiSeiKit-public's architectural claims.

## Files

| File | Role |
|---|---|
| [`PLAN.toml`](PLAN.toml) | **Hand-edited SSoT** — all claims with mechanical evidence |
| [`ARCH.md`](ARCH.md) | Generated by `kei-arch-map render` — module/claim map with hyperlinks |
| [`CLAIMS.md`](CLAIMS.md) | Generated by `kei-arch-map plan` — real-time PASS/FAIL audit |

## Workflow

1. **Edit `PLAN.toml`** to add/remove claims.
2. Run `cargo run -p kei-arch-map -- verify` to check.
3. Run `cargo run -p kei-arch-map -- render` to regenerate `ARCH.md`.
4. Run `cargo run -p kei-arch-map -- plan` to regenerate `CLAIMS.md`.
5. Commit all 3 files together.

## Why

Narrative claims in commit messages drift from cargo reality. Wave 2 audit
2026-05-04 caught a commit (`aaa8f36`) that asserted "VERIFIED: cargo check
exits clean" while cargo check actually failed. This SSoT mechanically
checks claims so a hook (Phase 2) can block commits where the claims and
the repo disagree.

## Schema

5 evidence kinds:

- `file_exists` — path exists on disk
- `regex_match` — regex matches in file content
- `exit_code` — `sh -c '...'` returns expected exit code
- `count_eq` — two `sh -c '...'` commands' stdouts (parsed as integers) are equal
- `http_status` — HTTP GET URL returns expected status code (live probe)

The conjunction `valid(state) = ∀c ∈ claims. c.holds(state)` is mechanically
checkable — that's the rigid schema hallucinations cannot break.
